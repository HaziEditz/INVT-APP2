Hi — full field map below. Two important context items first:

A. There are TWO trip-detail UIs on the dispatch console, and the dev who screenshot job #6112605163 was looking at the OLD one.

Old modal — #JobsDetailsSection, AngularJS scope showi.*, populated from the SQL stored proc JobDetails (backend ASP.NET, taxitime.co.nz). Field names are the PascalCase SQL column names from the legacy Booking table.
New modal — #job-detail-popup / ShowJobPopup(jobId), same SQL row PLUS a Firebase enrichment that reads from allbookings/{cid}/{bookingId} first, then falls back to completedJobs/{cid} searched by orderByChild('bookingId').equalTo(jobId) (i.e. push-keyed driver-app writes). This one already shows fare/timeline. If your dev points the dispatcher at "View → New job details" instead of the old PDF button, most of the "missing on screen" complaints disappear.
B. The PDF is dumb. It's doc.fromHTML($('#JobsDetailsSection').html()) of the old modal — that's why it has no layout and renders mojibake on ppname (jsPDF v1's fromHTML doesn't handle UTF-8). The PDF redesign is on me; you don't need to do anything for it.

All accept either ISO "YYYY-MM-DDTHH:mm:ss" (rendered verbatim as NZ local) or anything new Date() can parse. Use ISO with NO Z suffix and NO offset — the page treats unstamped ISO as already-NZ-local, which matches our TZ=Pacific/Auckland setup.

Old modal uses the SQL-side aliases on the same row: jobaccpettime, arrivedtime, pickuptime, onthewaytime, newcompelete, JobCompleteTime, DispatchTime. These are populated by the ASP.NET backend when driver actions come through /Login.svc proc calls; if you want them to appear when offline-synced, your syncOfflineTrip POST needs to land in the same SQL row (which it already does — confirmed working since v22au).

Firebase paths the dispatch page reads
Primary: allbookings/{cid}/{bookingId} — direct key lookup.
Fallback: completedJobs/{cid} .orderByChild('bookingId').equalTo(String(bookingId)).limitToFirst(1) — for your push-keyed completed records. (Note: this requires ".indexOn": ["bookingId"] on completedJobs/$cid — already in database.rules.json.)
Both nodes are merged onto the SQL row, with Firebase wins on non-empty values — meaning anything you write under the names above (PascalCase or the camelCase fallbacks I've listed) will populate the on-screen card and the PDF automatically.
